# Frontend Design Skill

当任务涉及新增、修改或重构前端页面、组件、交互、样式、前端状态管理或前端数据展示时，必须先使用本 skill 完成设计检查，再开始写代码。

## 目标

在实现前明确：用户目标、页面信息架构、关键交互、数据依赖、状态边界、响应式行为和验收方式，避免直接堆 UI 或把后端字段机械映射到界面。

## 必做流程

1. 确认用户场景：谁在什么时间使用页面，要完成什么决策或操作。
2. 梳理页面范围：新增/修改哪些页面、组件、路由或布局区域。
3. 定义信息层级：主指标、辅助信息、操作入口、风险提示的展示优先级。
4. 明确状态：加载、空数据、错误、部分数据缺失、权限不足、操作成功/失败。
5. 明确数据契约：页面需要哪些 API、字段、刷新时机、缓存或轮询策略。
6. 明确交互：筛选、排序、分页、搜索、表单校验、确认操作、撤销或重试。
7. 明确视觉约束：布局密度、响应式断点、图表/表格使用边界、可访问性。
8. 给出验收清单：实现后必须如何在真实页面验证 golden path 和边界状态。

## 输出格式

使用本 skill 后，先输出简短设计结论：

```markdown
## 前端设计结论
- 页面目标：
- 页面范围：
- 信息层级：
- 数据依赖：
- 关键交互：
- 必备状态：
- 验收方式：
```

如果任一项无法确定，先向用户提问；不要直接写前端代码。

## 策衡项目偏好

- 前端服务于交易辅助决策，优先展示“为什么值得关注/为什么要避免”，而不是只展示原始数据。
- 默认从日常工作流组织页面：数据同步、股票评估、策略选择、交易记录、复盘总结。
- 风险、置信度、数据更新时间和数据来源应靠近核心结论展示。
- 表格适合承载可扫描列表，详情页或侧边面板适合承载解释、证据和下一步动作。
